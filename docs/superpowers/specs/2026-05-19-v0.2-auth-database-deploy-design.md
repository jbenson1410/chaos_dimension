# Chaos Dimension v0.2 — Auth, Database, Public Demo, Deploy

**Status:** Approved
**Date:** 2026-05-19
**Target domain:** chaosdimension.fyi

## Context

v0.1 is a Vite + React SPA with in-memory state. v0.2 adds:

1. A public demo landing on `chaosdimension.fyi` so visitors can see what the tool is.
2. A login gate so the owner can manage their real board privately.
3. A database backend so tasks and agents persist.
4. A "fork and deploy your own" path that skips the demo and runs as a single-user instance for whoever clones it.
5. The schema needs to anticipate v0.3 MCP-based agent status reporting without rework.

The design choice is to keep the existing Vite SPA (no Next.js migration), add Vercel serverless functions for the API, and use Neon Postgres via Vercel's marketplace integration. SQLite was considered and rejected: Vercel serverless is stateless, so a local `.db` file would be wiped between invocations. Turso (SQLite-as-a-service) and Neon Postgres have equivalent "fork and click" deploy stories; Postgres is the more conventional pick.

## Architecture Overview

- **Frontend:** Vite + React 18 SPA, React Router for the new routes. Inline-styled Mac OS 7 chrome stays untouched.
- **Backend:** Vercel serverless functions in `/api/*`. No Express. Each route is one file.
- **Database:** Neon Postgres via Vercel marketplace integration. Drizzle ORM for schema + migrations. Migrations committed to repo.
- **Auth:** Single password env var (`CHAOS_PASSWORD_HASH`), bcrypt-compared. Session via `iron-session` (signed httpOnly cookie). No users table.
- **Mode toggle:** `CHAOS_PUBLIC_DEMO=true` enables the public demo landing on `/`. Default (unset) → `/` redirects to `/login`.

## Routes

| Route | Behavior |
|---|---|
| `/` | Demo mode: read-only board from `seed.js` + nav (GitHub, About, Login). Private mode: redirect to `/login`. |
| `/about` | Motivation, Parquet Courts reference, design notes. |
| `/login` | Password form. POSTs to `/api/login`. |
| `/app` | Authenticated app. Hits the DB. Redirects to `/login` on 401. |

## API

```
POST   /api/login          { password } → set session cookie
POST   /api/logout         clear cookie
GET    /api/me             200 if authed, 401 otherwise
GET    /api/tasks          list
POST   /api/tasks          create
PATCH  /api/tasks/[id]     update
DELETE /api/tasks/[id]     delete
GET    /api/agents         list
PATCH  /api/agents/[id]    update status / log
```

All mutation routes call `requireAuth(req)` → 401 if no valid session.

## Data Model

Single-tenant — no `user_id` columns. There is only ever one user per deployment.

### tasks
```
id                 text primary key (cuid)
title              text not null
workstream         text not null  -- FK to workstreams.id
column             text not null  -- 'backlog' | 'active' | 'review' | 'done'
agentDispatchable  boolean default false
priority           text default 'med'   -- 'high' | 'med' | 'low'
notes              text default ''
createdAt          timestamp default now()
updatedAt          timestamp default now()
```

### agents
```
id          text primary key
name        text not null
status      text default 'idle'   -- 'idle' | 'running' | 'error'
taskId      text references tasks.id on delete set null
startedAt   timestamp
log         jsonb default '[]'    -- array of strings
```

### workstreams
```
id     text primary key   -- 'second-seat', 'blog', etc.
label  text not null
color  text not null
icon   text not null
```

Workstreams seed from `src/data/workstreams.js` via `npm run db:seed`. Colors and icons stay versioned in code, not hand-edited in the DB.

## Auth Mechanics

- `CHAOS_PASSWORD_HASH` — bcrypt hash of the owner's password. Generated locally via `npm run hash-password`.
- `CHAOS_SESSION_SECRET` — 32-byte hex string. Used by `iron-session` to seal cookies.
- `POST /api/login` bcrypt-compares the submitted password. On match, seals a session payload `{ authed: true, iat }` into the cookie. Uses **`bcryptjs`** (pure JS) not native `bcrypt` — native bindings don't ship cleanly to Vercel serverless.
- Cookie: `httpOnly`, `Secure`, `SameSite=Lax`, 7-day expiry.
- `requireAuth(req)` util unseals the cookie and returns the session or throws 401.
- No registration. No password reset. Forgot password → re-set the env var.

## Demo Mode

- `App` component takes `mode: 'demo' | 'live'` prop.
- Demo: data sources from `seed.js`. Mutations are no-ops that show a Mac OS alert dialog: "This is a demo. Clone the repo to run your own."
- Live: data sources from `/api/*` with optimistic updates.
- Same component tree, swappable data layer.

## Deploy: chaosdimension.fyi

1. Push repo to GitHub.
2. Import in Vercel, attach custom domain `chaosdimension.fyi`.
3. Vercel dashboard → Add Neon Postgres (auto-injects `DATABASE_URL`).
4. Set env vars: `CHAOS_PUBLIC_DEMO=true`, `CHAOS_PASSWORD_HASH`, `CHAOS_SESSION_SECRET`.
5. Run migrations + seed: `vercel env pull && npm run db:push && npm run db:seed`.

## Fork Story (documented in README)

1. Fork repo, click "Deploy to Vercel" button.
2. In the new Vercel project, add Postgres via marketplace.
3. Locally: `npm run hash-password` → paste output as `CHAOS_PASSWORD_HASH`.
4. Generate `CHAOS_SESSION_SECRET` (`openssl rand -hex 32`), paste in Vercel env.
5. Leave `CHAOS_PUBLIC_DEMO` unset.
6. Run `npm run db:push && npm run db:seed`.
7. Visit deployment URL → see login page → enter password → use your own Chaos Dimension.

## Future: MCP (v0.3)

The v0.2 `agents` and `tasks` schema already carries the fields MCP will need: `agents.status`, `agents.log`, `agents.taskId`. A future MCP endpoint will be a thin write layer over the same tables — no v0.2 schema rework expected.

## Out of Scope (v0.2)

- WebSocket streaming for agent terminals (v0.3 Phase 2)
- File-based agent status watcher (`~/.chaos-dimension/agents/`) (v0.3 Phase 1)
- MCP endpoint (v0.3 Phase 3)
- Worklog export for blog content
- Multi-user support, password reset, registration, 2FA
- Drag-and-drop polish on the kanban (existing behavior preserved)

## Testing Strategy

- **Unit:** Drizzle schema queries against a local Postgres (or Neon dev branch) via `vitest`.
- **Integration:** API routes tested with `supertest`-style requests against a test Neon branch.
- **Manual:** Local `vercel dev` to verify demo/private mode toggle, login flow, and full task CRUD against a Neon dev branch. Verify Mac OS aesthetic is preserved across the new pages.
- **Production smoke:** Deploy to a preview URL first, smoke-test login + create/update task, then promote to chaosdimension.fyi.
